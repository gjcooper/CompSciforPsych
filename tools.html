Databases


Image courtesy of Pixabay
  • Structured Data
  • As soon as computers moved towards commercial use

Navigational Databases


Image courtesy of Pixabay
  • Data stored alongside it's relationships
  • Analogous to world wide web
  • Variations, heirarchical or network models
  • Quickly unwieldly
  • Navigate along multiple nodes to gather all data required

Relational Databases


Image courtesy of Pixabay
  • Data stored separate to relationships
  • Data stored in tables, linked to other tables
  • A lot of research and development into efficient storage and access of data
  • Anything that is very structured would ideally be stored in a RDBMS
  • MySQL, MariaDB, Postgresql, Microsoft SQL, Microsoft Access

NoSQL Databases


Image courtesy of Pixabay
  • Recent rise in need and desire to store less structured data
  • Designed over distributed storage
  • Four types
  1. Document Stores: MongoDB
  2. Column Based: Not focussed on rows
  3. Key-Value Based: Not useful for complex relations or when access via anything other than primary key
  4. Graph Databases: Storing social networks, based on graph theory

Source Control

  • Also known as version control or revision control
  • Manage changes to a set of data over time

First Generation


Image from Pixabay
  • RCS (Revision Control System), SCCS (Source Code Control System)
  • Concurrent development handled with locks
  • Only one person could work on a file at any one time
  • Networking not supported

Second Generation


Image courtesy of Pixabay
  • CVS (Concurrent Version Systems), Subversion, ...
  • Centralised Networking model
  • Allowed to make silmultaneous modifications
  • Merge current revisions into work before before committing
  • Multi-file operations (Commit changes to more than one file at a time)

Third Generation


Image courtesy of Pixabay
  • Distributed Version Control Systems
  • Git, Mercurial and Bazaar
  • Distributed Networking
  • Separates merge and commit
  • Operates on the level of changesets

Further comparison

  • A Centralised Version Control system requires central control
  • Decentralised VCS, everyone has a complete copy of the history

The rise of Git


Image courtesy of Pixabay
  • Created to manage the Linux Kernel Project
  • Meant to be fast and network efficient
  • Capable of handling large projects
  • Rise in large part because of the popularity of GitHub

Git Command Line Basics

> git init [directory]

> git clone [repo]

> git config user.name [name]

Other config options

  • core.autocrlf
  • core.editor
  • user.email
  • merge.tool

Use the --global option for all repos

Adding files to a repository

> git add [directory]

> git commit -v

> git commit -m [message]
  • git add one or more files
  • git add patch
  • git commit --amend

Viewing the state of the repo

> git status

> git log

> git diff
  • git log --graph --decorate
  • git log --author="[pattern]"
  • git log --grep="[pattern]"
  • Show glga

Undoing Edits

> git revert [commit]

> git reset [file]

> git clean -n
  • revert creates new commit to reverse
  • git reset --hard and git reset --soft
  • git clean -f (to force clean)

Branches

> git branch

> git checkout -b [branchname]

> git merge [branchname]

> git rebase [branchname]
  • Drop the -b for an existing branch
  • merge can be automatic or may have conflicts
  • rebase alters history, generally only for non shared code

Remote Repositories

> git remote add [name] [url]

> git fetch [remote] [branch]

> git pull [remote]

> git push [remote] [branch]
  • url can be local, ssh, https
  • for fetch leave off [branch] to get all remote refs
  • runs fetch followed by merge
  • sends local branch to remote
