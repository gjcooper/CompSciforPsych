High Level Languages

This covers a huge range of languages, technically everything else above Assembly.

Fortran

  • Developed in the 50's
  • One of the first programming languages
  • First version programs entered using punch cards
  • Formula Translation - for Numeric and scientific computation

Java


Image courtesy of Pixabay
  • Developed in the early 90's
  • Write Once, Run Everywhere
  • Heavily Object Orientated
  • Primary language of the Android Operating System

Python


Image courtesy of Pixabay
  • Developed in the early 90's
  • Design emphasizes readability (Tab/space indents are enforced
  • Dynamic (Duck) typing system
  • Supports Imperative, Procedural, Object Oriented and Functional Programming
  • Large and Comprehensive Standard library

MATLAB and R

  • Domain Specific Languages R(mid 70's) and Matlab(late 70's)
  • Matrix Laboratory is proprietary, primarily for numerical computing
  • R is open source and designed for statistical computing and graphing
  • A large array of domain specific library and addon code
  • Matlab written originally in Fortran and moved to C
  • R written in a mix of Fortran and C
  • Both use 1 based arrays/vectors compared to most C based programming languages

Programming Paradigms

  • Object Oriented, Procedural, Functional
  • Inevitably crossover betweeen different paradigms

Procedural Programming

  • Reusable code in the form of procedures
  • C and Fortran are early examples using this
  • Modules and scoping to keep workspace clean
  • Procedures have arguments and often a return value
  • Go is a recently developed language with this paradigm
  • Scoping prevents one procedure or module from having access to anothers variables
  • Scoping prevents conflicts and unintended consequences

Object Oriented Programming


  • Objects can contain data in the form of fields/attributes
  • Objects can also contain code in the form of procedures, often known as methods
  • C++, Java and Python are all significant users of OO programming to varying degrees
  • OO often class-based - but not necessarily so

Functional Programming

  • Treats computation as mathematical functions
  • Avoids mutable data and changing state
  • Output of a function depends only on inputs
  • R is an example of a language with a high degree of functional aspects

Basic Building Blocks

  • Arrays, Linked Lists, Stacks and Queues
  • Trees and Graphs
  • Dictionaries and Hashing
  • Arrays are indexed, fast access, fixed size, can be expensive to add elements to
  • Linked Lists - Linear time to access elements, fast to add elements
  • Stacks - LIFO, usually implemented on top of LL
  • Queue - LILO, usually implmented on top of LL
  • Trees - Many examples, Binary tree search O(logn)
  • Graphs - representing directed/undirected graphs from mathematics, edge weights etc
  • Dictionaries - Represent key/value pairs. Fast lookup and search

Greedy Algorithms

Make the locally optimal choice with the hope of finding a global optimum

For example the travelling salesman problem

Divide and Conquer

Break a problem down into smaller subunits that can be solved simply

Recursion is the way this is achieved, allowing a function to call itself

Quicksort is an example

  1. Pick a pivot value from the list
  2. Reorder the list so all values less than pivot a to it's left and all values larger are to the right
  3. Recursively apply steps 1 and 2 to the left and right sub-lists

Concurrent programming

An aside

Thread programming - Multitasking

  • Within the one process
  • Sharing state between threads
  • Difficult to reason around
  • Useful with network programming or user interaction
  • Lock files or message queues
  • Anything where you have to wait on a possibly long running process

Multiprocess programming

  • Multiple instances of the program
  • Communication is "external" to the program
  • Takes advantage of multiple cores
  • Cannot share state
  • Programming language may provide helper methods/abstractions
  • Multiple processor intensive tasks

Choosing a Language for your task

Considerations


Image courtesy of Pixabay
  • Many things to consider when picking what language to implement your task in.

Environment

  • Being run online?
  • Running on one or multiple OS's
  • Communicating with specialist hardware
  • Javascript for online
  • Different languages have varying degrees of platform specific code needed
  • Make sure you have all the requirements for your hardware gathered before starting

Experience

  • Have you extensive experience in one language?
  • Do you have access to experienced colleagues for assistance
  • Always weigh up briefly the suitability of the language vs the time to learn a new one
  • How much help can you get.

Available code

  • Has anyone got something similar in the lab
  • Is there anything published online you can use
  • Make sure you understand what is there, and assess how similar it really is
  • Same with online, something written in a language you are unfamiliar with could have unintended consequences.

Availability of Frameworks/Libraries

  • There are many powerful libraries/programs/frameworks out there
  • Does one fit you usecase well?
  • Python has Psychopy, Opensesame and Expyriment
  • Matlab has Psychtoolbox, Cogent, ...
  • Java has Tatool
  • Proprietary systems like Presentation, ePrime etc

Future reuse

  • Reuse by you?
  • Reuse by fellow researchers?
  • Posted on open science foundation or similar
  • Will you have access to underlying tech in future
  • Does anyone else use the language
  • Do any other researchers in the field use it?

Experimental Programming

Hardware Interfacing

  • Everything takes time
  • Measure that time if important
  • Minimise time delays
  • Monitors, keyboards, mice, other equipment
  • Time delay and jitter
  • Hardware solutions
  • Software solutions

Capturing all the data

  • When to record data?
  • What to record?
  • What format to record data?
  • What next?
  • As soon as possible without compromising timing
  • All the raw data, plus anything you calculate on the spot
  • Plain text logs, Structured data, Database
  • Backup your data, automated, manual, online or local

Programming Consideration

  • Timing
  • Flexible coding
  • Testing
  • Counterbalancing
  • Time sensitive parts must be given priority
  • Precalculate what you can
  • Create flexibility so that future changes are easier
  • Test your code, get someone else to test, and look at your code, especially sections which are critical
  • Counterbalancing instructions, response buttons, within or across subjects

Lab settings

  • MRI
  • EEG
  • Eyetracking
  • Prone position, no ferro-magnetic materials, narrow bore, limited time
  • Limit movement, limit electromagnetic interference, cap setup time
  • Lighting can be critical, limit movement, contrast on screen, contacts and glasses
